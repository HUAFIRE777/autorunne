# Autorunne 0.6.25 Handoff Cleanliness Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task if this becomes a coding slice.

**Goal:** Ship a small 0.6.25 patch that makes Autorunne handoff views cleaner after completed work, without changing the stable 0.6.24 entry flow.

**Architecture:** Keep the fix in the state/render/status layer. Add a small reusable sanitizer that removes completed/workflow-only items from the visible `next_product_task`, then call it during sync/render/status-producing flows so old repositories clean themselves on the next `autorunne open` / `sync` / `render`. Avoid broad workflow rewrites.

**Tech Stack:** Python, Typer CLI, pytest, existing Autorunne state files under `.autorunne/state/*`.

---

## Product scope

This should be a narrow patch, not a big feature release.

### Fix in 0.6.25
1. If `current.next_product_task` points to a task already in `tasks.completed`, user-facing views should not keep showing it as the next product task.
2. If `current.next_product_task` is workflow/release housekeeping such as “Use 0.6.24 as the current public baseline”, it should be shown as `workflow_follow_up`, not as product work.
3. `autorunne status` should label `.autorunne/` Git tracking clearly, so users do not misread `Tracked by git: False` as “this project is not a Git repo”.
4. Existing repos should self-heal on normal commands: `open`, `sync`, `render`, or `status` should not require users to manually edit `.autorunne/state/current.json`.

### Do not do in 0.6.25
- Do not change the working 0.6.24 auto-`git init` behavior.
- Do not redesign task storage.
- Do not add silent auto-upgrade.
- Do not make `.autorunne/` tracked by Git by default.

## Task 2: Implement next-slot sanitizer in state_engine

**Objective:** Centralize cleanup logic so old repos self-heal consistently.

**Files:**
- Modify: `src/autorunne/core/state_engine.py`

**Step 1: Add helper after `_first_pending_product_task`**

```python
def _completed_task_texts(state: dict[str, Any]) -> set[str]:
    return {
        (item.get("text") or "").strip()
        for item in state.get("tasks", {}).get("completed", [])
        if (item.get("text") or "").strip()
    }


def _clean_next_slots(state: dict[str, Any]) -> None:
    current = state.get("current", {})
    completed = _completed_task_texts(state)
    current_product = (current.get("next_product_task") or "").strip()
    current_next = (current.get("next_action") or "").strip()
    workflow_follow_up = (current.get("workflow_follow_up") or "").strip()

    workflow_candidates = []
    for candidate in (current_product, current_next):
        if candidate and _is_workflow_follow_up_text(candidate):
            workflow_candidates.append(candidate)
    if not workflow_follow_up or workflow_follow_up == "无":
        workflow_follow_up = workflow_candidates[0] if workflow_candidates else "无"

    excluded = [*completed, workflow_follow_up]
    product = current_product
    if not product or product in completed or _is_workflow_follow_up_text(product):
        product = _first_pending_product_task(state, exclude_texts=excluded)

    current["next_product_task"] = product
    current["workflow_follow_up"] = workflow_follow_up or "无"
    if workflow_follow_up and workflow_follow_up != "无":
        current["next_action"] = workflow_follow_up
    elif product:
        current["next_action"] = product
    else:
        current["next_action"] = current_next or "确认下一个具体任务"
```

**Step 2: Call helper before saving/rendering**

Add `_clean_next_slots(state)` in these places before `save_workspace_state(...)` / `render_views(...)`:
- `sync_workspace`
- `start_task` after `_set_next_slots` and task realignment is okay, but avoid overwriting a fresh active product next. If test risk appears, skip start.
- `record_checkpoint`
- `finish_task`
- `record_task_ingress`
- `render_from_state` path via `render_views` needs a clean-before-render route. Prefer adding cleanup inside `render_views(repo_root)` if available, or in `render_from_state` before rendering.

Implementation caution: if this helper touches `render_views`, avoid recursion. If `render_views` only reads state and writes views, update state once before rendering.
